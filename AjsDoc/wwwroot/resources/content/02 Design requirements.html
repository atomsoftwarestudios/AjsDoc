Design requirements

Before the decision was made and development of the framework begun some requirements were defined and other frameworks were searched to support them. Many of them partially supports them, but there is no framework fully fits requirements of the targetted project. The following list contains a few of them:

  • As short as possible learning curve
  • Single Page “Heavy” Client JS application framework
    • The application must be fully running on the remote device in browser, possibly offline.
  • As small as possible, ideally up to 150kB ungzipped
  • Browser support
    • IE10+ / Edge, Chrome, Firefox, Opera, Safari (including mobile versions)
  • Offline support
    • Full web applications or at least parts of them should be possible to run offline with a minimal efort during implementation
  • No requiremnt on server side rendering at all.
      Only code, resources and the data should be transferred over network. The application must be fully running on the remote device in the browser and should be able to render from templates itself
  • Complete code base
    • Not a collection of tons of independent libraries from various developers built together using some dependency loader
    • Support for 3rd party libraries
    • It should be possible to maintain it in-house
  • Well designed and developed, ideally with a good support
    • Separation of concerns and possibly of team roles
      • HTML should be HTML, ideally completely separated from the code and the view logic
      • Separation of layers based on MVC or MVVM
    • HTML / CSS / DOM
      • HTML templating must be easy. Ideally "coding" free (logicless) without tons of additional tags and attributes so web designers can use it without need of learning so much stuff.
      • Manipulation of DOM should not be necessary form the application (but possible)
      • From the application, nothing like createElement or $(… should be necessary - but of course possible
      • Frameworks / Libraries like jquery should be avoided but possible to use
      • Reusable HTML/CSS components
      • CSS frameworks should be usable with no impact
      • Stylesheets must be used at maximum level. Avoding manipulation of the style attribute is the big plus
      • DOM should be ideally manipulated only automatically based on the template and the data
    • Testable code
    • Typed language such as TypeScript
    • Build tool chain should be limited as much as possible (it would be possible to use Node, NPM and tons of tools, but better if it will not be necessary)
    • Complete and easy accessible debugging possibilities with break points and another nice stuff in some full featured IDE
    • Well documented and ideally supported
    • Ready for extensions and migrations (i.e. to ES6)
  • Browser must still work as a browser.
    • It’s still a web. So for the maximum possible User Experience keep it as a web.
  • It should be simple to use and optimized for rapid application development.
    • Reusability of templates and the view components code is a must
  • Easy language and translation support
    • It should be simple to translate the application to a different language.
  • As less as possible data transfers between the client and server
    • Data & and resource caching, ideally controllable from the application, is a must
